# Docs MDX Lab

The docs lab is the project-level workspace where Spectre documentation is read, annotated, and prepared for agent-assisted edits. The main surface is the rendered document, not a chat box. The user starts from the doc itself, highlights the specific passage or block that needs attention, and turns that selection into an actionable cue.

The lab keeps edits reviewable. A user cue becomes a queued action, the action can become a proposal, and the proposal is reviewed as both source diff and rendered preview before any file mutation is accepted.

## Behavior

The docs browser prefers MDX when an MDX sibling exists, while markdown files can remain in the corpus during migration. This lets the project move toward an MDX-native docs surface without losing the plain markdown comparison path.

The primary interaction flow is target-first:

```text
Rendered document
  -> hover or click a meaningful target
  -> select a block, text range, visual point, or custom element
  -> capture a stable anchor
  -> write an inline cue with intent
  -> queue a docs annotation
  -> record a docs action event
  -> propose an edit from queued context
  -> review source diff and rendered preview
```

The inline composer is the default authoring path for precise cues. The action pane is the companion surface: it shows document context, the current target, queued annotations, recorded action events, freeform fallback prompts, and review output.

## Data Shape

The docs lab uses a small action model as the shared route between the UI and docs agents.

| Concept | Shape | Purpose |
|---|---|---|
| Document | Project docs path, display path, source format, content hash, content | Identifies the editable artifact and protects proposals from stale reads. |
| Target | Kind, label, block id, block type, optional source range, optional visual coordinates, optional custom element id | Identifies what the user is pointing at in the rendered document. |
| Anchor | Document path, content hash, selected quote, optional source range, context before/after, optional block id, target kind, target metadata | Binds a user cue to a specific passage, structured block, visual point, or custom element. |
| Annotation | Intent, body, anchor, status, resolution target, author, timestamps | Represents the user's requested change, question, deletion, or note. |
| Action event | Event type, actor, document, related annotation or proposal, payload, timestamp | Records the user-agent interaction stream for the docs lab. |
| Proposal | Source annotations, prompt, generated content, original hash, proposed hash, diff, rendered preview | Stages an edit for review before apply. |

The action ledger is Spectre's docs-side equivalent of an agent-native action channel. It is not the session plan. `plan.json` remains the workflow execution primitive; docs action events are the document-editing primitive that lets the UI and future docs agents share intent, context, and review state.

## Custom Blocks

MDX blocks give the docs lab structured editing targets inside otherwise prose-heavy documentation. Each supported block type has a stable tag, semantic type, target kind, label, agent-facing description, and patch operation vocabulary.

The initial block vocabulary is:

| Block | Role |
|---|---|
| `Decision` | Captures accepted, proposed, or rejected project decisions with durable rationale. |
| `Callout` | Highlights warnings, notes, tips, and other local guidance. |
| `AgentContract` | Describes a boundary that an agent should obey or satisfy. |
| `FileTree` | Renders a file hierarchy as a structured docs block. |

The document/support slice adds read-only blocks for review checklists, structured tables, tabbed comparisons, compact column layouts, and code snippets with source metadata.

Agents should treat registered blocks as addressable units. When a selected passage belongs to a block, the anchor carries the block identity and target kind so the proposal can reason about the right operation instead of treating every edit as raw prose replacement.

## Contracts

- A selected cue must preserve enough target context to be useful after the selection disappears from the screen.
- Text selection and rendered target selection are both first-class. A user may drag across exact words or click a whole heading, paragraph, list item, code block, table, or custom MDX block.
- A proposal must include both the textual source change and the rendered result.
- A docs agent must stage proposals through the ledger instead of mutating docs files directly.
- A block-aware proposal should prefer the block's declared patch operations over unstructured string rewrites.
- The action pane may create freeform requests, but precise document edits should start from an inline anchor whenever possible.

## Boundaries

The docs lab owns project documentation browsing, annotations, action events, custom block previews, and staged docs proposals.

The session workflow owns intake, spec, plan, build, docs update, and finalization phases. The docs lab can feed future docs agents with anchored context, but it does not replace session state, plan checkpoints, or end-of-session docs update decisions.

Durable agent-kernel integration requires a persistent ledger and proposal lifecycle. The current model defines the UI-agent contract first: anchors and action events are the stable primitive, while autonomous proposal generation and apply/reject workflows can attach behind that contract.
