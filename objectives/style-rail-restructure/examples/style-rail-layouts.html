Style rail layout explorations

Five layouts for the style sidebar, rendered at the proposed width. Shared premise: it stays a docked, flag-off-able panel — wider, calmer, and organized so "we're looking at callouts, adjust them" is a two-click thought. Direction: Apple simplicity with Unity-inspector density where it earns its keep.

Width: 36rem fixed576px, up from 30rem / 480px. No drag handle. Every mockup below is rendered at this width.
Still a sidebarDocked right, pushes the doc, removable via theme-lock flag. No takeover mode.
One control anatomyAll five variations share the same rows, swatches, sliders, switches — only the navigation differs.
Override dotsA blue dot marks any token changed from theme default — visible at every level, borrowed from Unity/Xcode.
A

Two-pane rail

macOS System Settings, in miniature. A persistent category rail on the left; the selected category's controls fill the rest. Nothing is ever more than one click away, and the rail doubles as a map of the whole theming surface.

How it works

  • Rail groups: Theme (globals) and Blocks (the 18 component overrides), one scroll.
  • Override dots in the rail show where you've diverged from the theme — the map shows state, not just names.
  • Detail header states the layering: "3 overrides · layered over Default theme."

Wins

  • Zero burying — every destination visible at once.
  • Fastest block-to-block hopping (Callout → Quote is one click).
  • The closest fit to "Apple and Unity had a baby": Settings chassis, inspector contents.

Costs

  • Rail eats ~170px; detail pane gets ~400px — sliders fit, but it's the tightest variation.
  • 18 block entries make the rail itself scroll.

How it works

  • Every section is a foldout with a token count and override count in the header — closed sections still tell you their state.
  • The filter field searches token names across all sections: type "radius" and only radius rows remain.
  • Compact 30px rows; controls inline, right-aligned.

Wins

  • No modes, no navigation stack — Cmd-F thinking.
  • Filter is the killer feature: cross-cutting edits ("all radii", "all accents") that no other layout can do.
  • Most information per pixel; genuine Unity DNA.

Costs

  • Least Apple of the five — density reads as "engine editor," not Settings.
  • With many foldouts open it degenerates back into today's long scroll.
D

Specimen-first editor

Built around the sentence you said: "we are looking at callouts." A live specimen of the selected block is pinned at the top of the panel, controls sit under it, and a chip strip switches blocks. The panel becomes a little workbench for one block at a time.

How it works

  • Three segments: Theme (globals), Blocks (this view), Layout.
  • The specimen is a real render of the block on a neutral artboard — it moves as you drag sliders, even if no callout is on screen in the doc.
  • Chips scroll horizontally; the active block's token file is named in the footer of the specimen.

Wins

  • Directly answers "we're looking at callouts" — subject and controls share one frame.
  • You can style a block the current doc doesn't contain.
  • Tightest feedback loop of the five; feels like a product, not a settings page.

Costs

  • Specimen costs ~140px of vertical space forever.
  • Chip strip hides blocks past the fold; 18 chips need overflow handling.
  • Needs per-block specimen fixtures — real build cost.

How it works

  • Each card is the block rendered tiny with the current theme — restyle Code dark and its thumbnail goes dark.
  • Selecting a card pins its quick-edit rows below the grid; "All properties…" drills into the full set.

Costs

  • Grid + quick edits + full drill-in = three layers of UI.
  • Slowest to a specific slider if you already know what you want.

Round 2

A × D, fixed for a right-docked panel

Feedback from round 1: A and D won — the two panes, the clean sliders, the preview colors. The flaw: A puts its rail on the panel's left, so when the panel docks right, the navigation lands mid-screen, wedged between the doc and the controls it drives. Three fixes below, each shown next to a ghost of the doc so the adjacency reads. All three keep D's specimen.

F

How it works

  • Same rail as A — Theme + Blocks groups, override dots — moved to the panel's right edge, flush with the screen.
  • Detail pane sits adjacent to the doc; the specimen strip (~90px) is the compact form of D's, collapsible.
  • Selection flow reads outside-in: edge rail → controls → live doc.

Wins

  • Keeps everything you liked in A: visible map, labels, one-click block hopping.
  • Screen-edge rail is the easiest target on the screen to hit (edges have infinite depth for the cursor).
  • Controls and doc are neighbors — tweak-and-watch happens in one eye span.

Costs

  • Rail-right is less conventional; first-run it reads slightly RTL.
  • Detail pane stays ~400px — same as A, fine for current controls.

How it works

  • Icon rail carries only the six top-level categories: Presets, Colors, Typography, Blocks, Background, References. Tooltips name them; dots mark overrides.
  • Blocks category is pure D inside: chips, specimen, full-width rows (~510px of control space).

  • Six icons must be learned; tooltips carry the load at first.
  • Blocks return to chips — 18 chips overflow, so block hopping regresses vs F's list.
H

How it works

  • Menu rows carry glyphs and override counts — the status map from A's rail, on demand.
  • Search inside the menu; typeahead selects.

  • Block hop = two interactions (open, choose). Worst of round 2 for browsing.

Recommendation — build F

F keeps everything that made A win (the always-visible map, labels, one-click block hopping) and everything that made D win (the specimen, the tight feedback loop), while the mirrored rail dissolves the docked-right awkwardness — navigation lives on the screen edge, controls sit beside the doc. If real controls make F's ~400px detail pane feel tight, G is the pressure valve: same skeleton, rail collapses to icons. H's searchable picker menu is worth keeping in the drawer for when the block count outgrows a rail.

1 · ChassisWidth 30rem → 36rem; StyleRail becomes mirrored two-pane (detail left, rail right). Rail groups: Theme / Blocks.
2 · State surfacingOverride dots + counts everywhere. The token registry already knows defaults, so "diverged" is computable today.
3 · Rail previewsSwatch strip on Colors, "Sans · 16px" on Typography, override counts on blocks — rows carry their values.
4 · Specimen stripCompact live render above a selected block's controls. Needs per-block fixtures — ship after the chassis lands.
